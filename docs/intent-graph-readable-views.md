# Intent Graph — Human-Readable Views

## Summary

The intent graph already contains the semantic content needed to
produce human-readable descriptions of what the business intends and
what actually happened during execution. Both views are generated on
demand by the LLM reading the existing graph structure. No parallel
text layer is maintained.

---

## Two Views, One Source

### Intents View

A readable description of what the business has decided should happen,
generated by the LLM traversing a subgraph and reading human-facing
metadata fields on each node.

Triggered by questions like:
- "Show me what the onboarding process does"
- "What is the qualifying analysis supposed to produce"
- "Walk me through the managed service monitoring loop"

The LLM reads the relevant subgraph and generates a coherent natural
language account. The human reads the account, verifies it reflects
their intent, and either confirms or asks the LLM to update the graph.

### Execution Trace

A narrative account of what actually happened during a specific
invocation, generated by the LLM correlating events table entries
against the relevant intent graph nodes.

Triggered by questions like:
- "Walk me through what happened with the Henderson engagement"
- "Why did the qualifying analysis fail"
- "What artifacts did the migration proposal produce"

The LLM reads the events table records for the invocation, maps them
against the graph nodes, and generates an account of what happened
relative to what was intended. Detail level varies with the question.
Raw events table entries are not surfaced unless specifically requested.

---

## What Keeps the Views Consistent

Neither view is stored. Both are generated from the current graph state
and the current events table. There is no synchronization problem
between a human-readable layer and the underlying data because no
separate layer exists.

When a graph node is updated, the next intents view generated from it
reflects the update automatically. When a new events table entry is
written, the next trace generated from it includes it automatically.

## Schema Addition Required

The current graph node records need additional human-facing metadata
fields. These fields are ignored by the runtime. They are read by the
LLM when generating either view.

### Fields to Add to Node Schema

```json
{
  "id": "node_ref",
  "type": "existing_type",

  "label": "Short human-readable name for this operation",
  "purpose": "What this operation is intended to accomplish, in business terms",
  "owner": "role_ref | agent_ref | null",

  ... existing runtime fields unchanged ...
}
```

### Fields to Add to Artifact Schema

```json
{
  "id": "artifact_ref",
  "type": "existing_type | document | decision | conversation_record",

  "label": "Short human-readable name for this artifact",
  "description": "What this artifact represents in business terms",

  "produced_by": "node_ref | conversation_ref",
  "content_ref": "uri | events_table_ref",
  "consumed_by": ["node_ref"],

  ... existing fields unchanged ...
}
```

The `type` field on artifacts gains new values to accommodate non-code
artifacts produced by human-LLM conversations. See artifact extension
document for detail.

---

## What the LLM Does With These Fields

### Generating an Intents View

1. Receive a human question referencing a process, operation, or intent
2. Identify the relevant subgraph — the entry node and its downstream
   dependencies
3. Traverse the subgraph, reading `label`, `purpose`, `owner`, and
   artifact `label` and `description` fields
4. Generate a natural language account of the process in business terms
5. Present to human for confirmation or correction

If the human identifies an error — "that's not quite right, the
diagnostic report also goes to the client before we proceed" — the LLM
updates the relevant graph node and regenerates the view.

### Generating an Execution Trace

1. Receive a human question referencing a specific invocation,
   engagement, or outcome
2. Identify the relevant graph nodes
3. Query the events table for entries associated with those nodes and
   the specified invocation
4. Correlate events table entries against graph node intent — what was
   expected vs. what the record shows
5. Generate a narrative account at the appropriate level of detail
6. Surface discrepancies between intent and execution if present

---

## What Does Not Change

- Graph structure
- Node types
- Edge types
- Expression evaluator
- Runtime traversal and execution logic
- Events table schema
- LLM read/write interface to the graph

---

## What Needs to Be Built

### 1. Schema Extension

Add `label`, `purpose`, and `owner` fields to the node schema
definition. Add `label`, `description`, `produced_by`, and
`content_ref` to the artifact schema definition.

These fields should be optional with null defaults so existing nodes
remain valid without modification.

### 2. Subgraph Query Interface

A query that takes a node reference and returns the subgraph rooted
at that node — the node, its downstream dependencies, and the
artifacts connecting them. Used by the LLM to retrieve the relevant
context for either view without loading the full graph.

```
get_subgraph(node_id, direction: downstream | upstream | both, depth: n | full)
```

### 3. Invocation Query Interface

A query that takes a node reference and an invocation identifier and
returns the events table entries associated with that invocation.

```
get_invocation_events(node_id, invocation_id)
get_invocation_events(node_id, time_range: {from, to})
```

### 4. LLM System Prompt Extension

The LLM's operational context needs to include:
- The convention for reading `label` and `purpose` fields when
  generating human-facing descriptions
- The convention for correlating events table entries against graph
  nodes when generating traces
- The instruction to present intents view output for human confirmation
  before treating it as verified

---

## Prototype Path

1. Add `label` and `purpose` fields to three or four existing nodes in
   the Northwind experiment graph — pick nodes that form a complete
   small subgraph
2. Write a prompt asking the LLM to generate a human-readable
   description of that subgraph using only those fields
3. Evaluate whether the output is what a business owner would
   recognize as an accurate description of the process
4. Identify what the LLM reaches for that isn't in the fields — that
   gap defines any additional schema additions needed

The subgraph query and invocation query interfaces can be built once
the schema addition is validated. The LLM system prompt extension
follows from what the prototype reveals about what the LLM needs to
generate reliable output.
